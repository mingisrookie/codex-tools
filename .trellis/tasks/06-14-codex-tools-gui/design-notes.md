## 总体设计原则

- 借鉴视觉结构，不照抄 UI。
- 优先提升信息层级、可扫读性和排障效率。
- 不引入官方产品线功能：不做 Analytics、CLI/TUI/npm、Anthropic 协议兼容、多 API Key 管理 UI。
- 图片稿只作为方向参考，最终实现以 React/Tauri 当前组件、类型、i18n 和样式约束为准。
- 2026-06-14 实机反馈后追加降噪方向：减少大面积渐变、玻璃和阴影；保留顶部单导航，移除底部重复 dock；页面更接近浅色企业控制台而不是营销首屏。
- 2026-06-14 第二版降噪方向：首屏应更像日常运维表格/详情面板，顶部只保留单导航和薄指标条，按钮、边框和状态色保持克制。

## 账号页

可采用：

- 顶部统计卡：账号数、活跃账号、已耗尽、异常账号。
- 主列表 + 右侧详情 inspector。
- 列表行内使用 5h / 1week 用量条、状态 chip、切换按钮、启用开关。
- 右侧详情展示：账号详情、使用概览、重置时间、最近切换记录、快捷操作。

2026-06-14 已实现：

- 账号页主列表从大卡片改成“表格列表 + 右侧 inspector”。
- 表格一行一个可切换账号 variant，保留当前账号、启停用、授权异常、用量异常和反代可用性状态。
- 右侧 inspector 承接原卡片快捷操作：切换并启动、停用/启用、导出、重新授权、设置别名、二次确认删除。
- 继续保留敏感账号 label 脱敏展示；长 endpoint、model、错误文案在详情区折行，不撑破布局。
- 小屏下表格与 inspector 变为上下布局，表格内部横向滚动。

避免：

- 不做分页复杂化，除非账号数量确实很大。
- 不暴露完整账号邮箱、account id、token 或 API key。

## API 反代页

可采用：

- 顶部状态总览：运行状态、端口、模型数量、当前账号池、健康状态。
- 分区卡片：本地代理配置、API Key、账号策略、平均负载、Cooldown 状态、远程访问、高级设置。
- 会话粘性作为默认开启能力展示在策略卡中。

避免：

- 不做完整多 API Key 管理。
- 不把本页做成产品化营销页；重点仍是运维控制台。

## Dashboard 页

可采用：

- 顶部指标卡：请求数、失败率、P95 延迟、Token 消耗。
- 时间线图：请求量 + 失败叠加。
- 最近失败列表：错误类型、分类、摘要可展开。
- 路由解释表：来源、模型、命中账号、候选数、affinity、cooldown、latency 选择原因。
- 过滤器：时间范围、模型、来源、账号。

避免：

- 不只做漂亮图表；必须能解释真实失败和路由选择。
- 不在 UI 中泄露敏感账号、token、key。

## 设置页

可采用：

- 设置分区：启动与后台、更新、编辑器联动、外观、高级。
- 右侧系统状态卡：数据目录、Codex CLI、当前版本、自动更新状态。
- 高级危险操作独立红色按钮，避免误触。

避免：

- 不把所有设置堆成一长列。
- 不显示真实完整本地私密路径；必要时截断。

## 实现顺序建议

1. 账号页：最容易感知 GUI 提升。
2. Dashboard 页：配合后端路由解释增强。
3. API 反代页：配合 session affinity 和状态解释。
4. 设置页：最后做整理，不阻塞核心代理能力。
