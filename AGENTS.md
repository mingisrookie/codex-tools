# Codex Tools 项目规则

本文件是给 Codex/AI 代理的仓库级执行入口。任何涉及本仓库代码、脚本、配置、文档、提交、发布的任务，都先按这里执行。

## 开发前必读

开始动手前必须实际阅读并以当前文件内容为准：

1. [项目文件结构说明.md](./项目文件结构说明.md)
2. [项目完整链路说明.md](./项目完整链路说明.md)
3. [项目开发规范（AI协作）.md](./项目开发规范（AI协作）.md)
4. 如果涉及 PR / 推送 / 合并，再读 [开发者AI开发与PR提交流程.md](./开发者AI开发与PR提交流程.md)

不能只凭历史记忆、会话摘要或旧模板直接开发。

## 本仓库架构基线

- 前端：React 19 + TypeScript + Vite，入口是 `src/main.tsx`、`src/App.tsx`，集中控制 hook 是 `src/hooks/useCodexController.ts`。
- 桌面后端：Tauri 2 + Rust，命令注册在 `src-tauri/src/lib.rs`，业务模块位于 `src-tauri/src/*.rs`。
- 本地 API 反代：Axum 服务在 `src-tauri/src/proxy_service.rs`，独立 daemon 入口是 `src-tauri/src/bin/codex-tools-proxyd.rs` 和 `src-tauri/proxyd/`。
- 长期说明文档以根目录三份中文文档和 `docs/` 为准；实现改变后必须同步长期文档，不用临时草稿替代。

## gpt-image-2 / image2 调用

当生成或编辑图片时，无论使用 `gpt-image-2`、image2、OpenAI Images API、`image_gen`，还是 OpenAI-compatible image client，默认都必须走 Codex Tools 本地 API 代理；除非用户明确要求绕过本地代理。

- Base URL：`http://127.0.0.1:<Codex Tools API proxy port>/v1`，当前默认端口是 `8787`
- 生成端点：`/images/generations`
- 编辑端点：`/images/edits`
- 模型：`gpt-image-2`
- API key：以 Codex Tools API 反代面板显示的本地 proxy key 为准

不要静默直连 `api.openai.com` 调用 `gpt-image-2`。如果本地代理端口或 key 不确定，先检查 Codex Tools 本地状态、配置或面板；仍无法确定时，向用户索要本地端口/key。

## 开发硬要求

- 先读真实代码链路，再改；优先用 `rg`、定向文件阅读和现有调用链确认运行事实。
- 不把新逻辑堆进大文件；前端状态优先收敛在 hook/类型/组件边界内，后端能力优先收敛在对应 Rust service 模块内。
- 账号、认证、API 代理、cloudflared、远程 proxyd、设置、更新、i18n 任一链路变化，都必须同步检查前端展示、Tauri 命令、Rust 业务模块、持久化、文档和验证命令。
- 文档和中文文案修改后必须做乱码检查；出现异常替换字符（U+FFFD）、错码、冲突标记或失效链接，任务未完成。
- 提交或推送前必须确认 `git status --short` 只包含本次任务相关文件，并运行与改动面匹配的检查。

1. 使用 `dxm` skill。

   - `项目开发规范（AI协作）.md`
   - `项目完整链路说明.md`
   - `项目文件结构说明.md`
   - `开发者AI开发与PR提交流程.md`
3. 已存在的人工维护文档不得静默覆盖；只补缺失文件或在用户明确要求时覆盖。
4. 生成或补齐后，本目录后续工作必须默认遵守这些规则。

当用户输入 `/dxm trellis`、`/dxm 大开发`、`/dxm full`，或明确说要启用“大开发工作流”时：

1. 先执行标准 `/dxm` 检查，确保上述长期文档存在。
2. 再启用 Trellis/Codex 项目工作流：创建或补齐 `.trellis/`、`.codex/`、`.agents/skills/`。
3. 已存在的 Trellis、Codex、AGENTS 或 DXM 文档不得静默覆盖；只能补缺失项或在用户明确要求时覆盖。
4. 如果当前目录不是项目根目录，先切到真实项目根目录，不要在宽泛盘符根目录误初始化。

## 大开发工作流约定

中大型开发、新模块、架构改造、跨多文件重构、多阶段任务或需求不清楚的任务，默认走：

1. 默认用 `grill-with-docs` 先把需求问清楚，并要求它尽量批量提问、能从上下文判断的就自行拿主意。
   - 只有用户明确要求，或完全没有可用项目/文档上下文且用户想轻量 brainstorm 时，才用 `grill-me`。
   - 对是否、命名偏好、低风险取舍这类问题，AI 应给出推荐判断并作为假设推进；只把真正阻塞、高影响、无法从代码/文档推断的问题汇总给用户拍板。
2. 将结论落盘到 `.trellis/tasks/<task>/prd.md`，不能只停留在聊天上下文里。
3. 用 Trellis 进入任务执行：implement → check → update-spec → finish。
4. Trellis 的 spec/journal 只负责任务记忆；长期项目事实仍必须同步回 DXM 根目录文档。

小修小补、只读分析、单文件轻维护不强制上 Trellis，除非用户明确要求“大开发模式”。

## 开发前必读

任何涉及代码、测试、脚本、配置、文档、提交、推送、PR 或发布的任务，开始前必须实际阅读或重新核对：

1. `项目文件结构说明.md`
2. `项目完整链路说明.md`
3. `项目开发规范（AI协作）.md`
4. 如涉及 GitHub / PR / 合并，再读 `开发者AI开发与PR提交流程.md`

不能只凭历史记忆、上次会话摘要或“看起来知道项目”直接开发。

## 意图边界

- 用户说“只分析 / 先看看 / 暂时不改”时，只能做只读分析，不得擅自改文件、提交或推送。
- 用户说“开始开发 / 直接改 / 按清单开发 / 提交代码”时，必须按阶段推进到可验证完成，不能停在建议层。
- 如果需求、目标、影响范围不清楚，先用最少问题澄清；本地能查清的，不要反问。

## 工作方式

- 先用真实文件、命令输出、日志、测试、diff 和运行态证据建立结论。
- 新功能必须沿现有分层接入，不能把逻辑重新堆回主入口或无关模块。
- 修改配置、状态、流程、UI、日志、恢复、测试、文档时，要检查完整链路而不是只改表面调用点。
- 每完成一个阶段要自检：影响文件是否完整、状态流/日志流/文档是否一致、是否引入乱码、定向检查是否通过。
- 最终汇报前必须做全局审查：工作区范围、测试/静态检查、文档同步、乱码检查、残余风险。

## 文档同步

- 新增、删除、重命名或改变文件职责：更新 `项目文件结构说明.md`。
- 功能链路、运行模式、状态流、输出文件、故障归因变化：更新 `项目完整链路说明.md`。
- 开发流程、架构边界、测试要求、协作约束变化：更新 `项目开发规范（AI协作）.md`。
- 不能只在临时方案文件里记录最终事实；长期事实必须回写根目录长期文档。

## 编码与敏感信息

- 中文文档、注释、日志、UI 文案、错误提示出现乱码视为未完成。
- 修改中文内容后必须显式检查可见乱码、错码、异常替换字符。
- 不得在回复或文档中回显真实 token、密码、API Key、账号明细、验证码、会话或密钥文件内容。

## 最终回执

完成开发或审查后，最终回复必须简明说明：

1. 改了什么。
2. 是否遵守本规范，尤其是架构边界、文档同步、测试和乱码检查。
3. 跑了哪些测试或检查。
4. 是否提交、提交号是什么。
5. 是否推送、推送到哪个分支。
6. 未完成项、未运行测试或残余风险。

不能只回复“已完成”，也不能隐藏失败测试、未跑测试、未更新文档。
